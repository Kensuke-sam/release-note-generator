# Interview Guide: release-note-generator

## 30秒サマリー

- デプロイ情報から、共有用リリースノート草案を生成する API。
- 配布文面と元データを分離し、レビューと再生成をしやすくした。
- 開発者体験だけでなく、運用・ロールバック観点を含めた設計。

## 3分説明の順番

1. 課題設定: リリース内容共有が遅い/属人化する。  
2. データ設計: デプロイ記録とノート草案を分離し、監査可能にした。  
3. AI設計: 出力を構造化して、Slackや社内ポータル連携しやすくした。  
4. 品質: Docker、CI、型チェックで継続開発しやすい土台を作った。  
5. 拡張余地: 変更差分自動取得、配布先別テンプレート、承認フロー。

## 深掘り質問に対する答え方

- なぜ草案として扱うか: リリース文は対外影響があるため最終レビューが必要だから。  
- なぜロールバック項目を持つか: 変更内容と運用リスクを同時に確認できるため。  
- 学生として何を見せるか: 「便利さ」だけでなく「安全性」を意識した設計判断。
